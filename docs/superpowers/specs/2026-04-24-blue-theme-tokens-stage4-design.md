# Blue Theme Tokens Stage 4 Design

## Context

Stage 3.6 introduced a coherent visual layer, but the palette is still hard-coded as a warm paper theme. Review showed two important follow-ups:

- Page background must be a true page-level background, not only an inner shell wrapper background.
- The visual system should support light and dark color schemes rather than a single fixed palette.

The selected Stage 4 direction is **Fuwari-ish Air Blue + Midnight Blue Dark**. Light mode should feel close to white with a slight blue cast. Dark mode should feel close to black with a deep blue cast. The intent is to match the mood of Fuwari's soft blue theming without copying Fuwari's implementation, exact colors, or Astro/Tailwind structure.

## Goals

- Replace hard-coded warm visual colors with a small blue theme token layer.
- Support light and dark visual output through Raptor's public `EnvironmentConditions.colorScheme` mechanism.
- Keep the page background close to pure white in light mode and close to pure black in dark mode, with only a subtle blue tint.
- Keep existing Stage 3 shell layout, sidebar behavior, route structure, and content model unchanged.
- Make future visual work consume semantic tokens instead of scattering raw color literals across styles.

## Non-Goals

- No explicit light/dark toggle button.
- No custom JavaScript color scheme controller.
- No route changes.
- No new content metadata.
- No article redesign.
- No search, table of contents, hero image, or animation layer.
- No exact copy of Fuwari colors, CSS classes, components, or layout.
- No attempt to solve Raptor generated CSS duplication.

## Visual Direction

Light mode uses the selected **B. Fuwari-ish Air Blue** direction:

- Page background: nearly white with a cool blue tint.
- Surfaces: white or blue-white cards.
- Borders: low-saturation pale blue.
- Text: dark navy rather than neutral black.
- Muted text: blue-gray.
- Accent: calm medium blue for links or emphasis.
- Shadows: soft blue-tinted shadows.

Dark mode uses the selected **C. Midnight Blue Dark** direction:

- Page background: near black with a deep blue tint.
- Surfaces: dark blue-black cards.
- Borders: subdued blue lines.
- Text: pale blue-white.
- Muted text: desaturated blue-gray.
- Accent: lighter blue for links or emphasis.
- Shadows: dark, low-glow shadows that avoid a gray dashboard feel.

The design should remain quiet and content-first. The palette should make the site feel airy, readable, and personal, not neon, cyberpunk, or dashboard-like.

## Architecture

Add a small theme token layer under a dedicated source area:

- `Sources/Theme/SiteThemePalette.swift`

The palette should expose semantic color roles, not component-specific names:

- `pageBackground`
- `canvasBackground`
- `surface`
- `surfaceRaised`
- `border`
- `text`
- `mutedText`
- `accent`
- `shadow`

Use two static palettes:

- `SiteThemePalette.light`
- `SiteThemePalette.dark`

Existing styles should consume these palettes rather than raw color literals:

- `PageCanvasStyle`
- `ContentSurfaceStyle`
- `PostCardStyle`
- `MetadataTextStyle`
- `SidebarPanelStyle`
- `MainLayout` page background

Shell layout styles should stay separate from theme tokens. `SiteShellStyle`, `ShellMainStyle`, and `ShellSidebarStyle` should not become color owners unless they already need a semantic color for their own visual responsibility.

## Raptor Color Scheme Strategy

Use Raptor's public style environment:

```swift
func style(content: Content, environment: EnvironmentConditions) -> Content
```

For visual styles, choose the palette based on:

- `environment.colorScheme == .dark` uses `SiteThemePalette.dark`
- all other cases use `SiteThemePalette.light`

This should allow Raptor's CSS generator to emit conditional dark-mode CSS through its existing `data-color-scheme="dark"` handling.

The site should not introduce custom color scheme state in Stage 4. Raptor's existing automatic color scheme support remains the source of truth.

## Page Background

`MainLayout` should continue to set page background with Raptor's public `Main.background(_:)` API so the page-level `--bg-page` variable reaches the document root output.

Stage 4 should move the concrete light page background color into the token layer. If Raptor's `Main.background(_:)` only accepts a single color and cannot vary by color scheme from public API, Stage 4 should:

- keep the light page background registered through `Main.background(_:)`
- use CSS generated by `PageCanvasStyle` and other visual styles for dark surfaces
- add tests documenting this limitation

Do not invent private API access to force page background switching.

## Component And Style Changes

`PageCanvasStyle` should represent page-level breathing room and canvas color. It should use light and dark token values through `EnvironmentConditions.colorScheme`.

`ContentSurfaceStyle` should style reusable cards with tokenized surface, border, and shadow values.

`PostCardStyle` should style post list item rhythm and primary text color through tokens.

`MetadataTextStyle` should style date and description metadata with tokenized muted text.

`SidebarPanelStyle` should align sidebar cards with the same surface, border, text, and shadow tokens.

`PostListItem`, `PostMeta`, route pages, taxonomy pages, and shell marker ownership should not change in Stage 4 unless required by token adoption.

## Testing

Stage 4 tests should prove the token migration changes behavior, not only code shape.

Publishing tests should check:

- generated HTML still includes page canvas, post card, metadata, and sidebar style classes
- generated HTML still includes the page-level `--bg-page` light background variable
- generated CSS includes light blue token values
- generated CSS includes dark blue token values
- generated CSS includes selectors or media output tied to dark color scheme handling
- generated CSS still avoids `@media (min-width: 0px)` regressions for visual style classes

Existing route coverage should remain broad:

- home
- archive
- about
- article
- tag and category routes through existing suites

Run:

```bash
swift test
```

Publish and inspect generated output:

```bash
swift run RaptorTsubame
rg -n "page-canvas-style|post-card-style|content-surface-style|metadata-text-style|sidebar-panel-style|data-color-scheme|--bg-page" Build/index.html Build/css/raptor-core.css
```

## Acceptance Criteria

- The hard-coded warm palette is replaced by blue semantic theme tokens.
- Light output uses a near-white air-blue palette.
- Dark output uses a near-black midnight-blue palette.
- Existing shell structure, marker ownership, sidebar behavior, and routes remain unchanged.
- Visual styles choose token values through public Raptor APIs.
- Published CSS proves both light and dark token values are emitted.
- Published HTML proves page-level light background registration remains present.
- `swift test` passes.
- `swift run RaptorTsubame` publishes successfully.

## Residual Risks

- Raptor may not expose public API for color-scheme-specific `Main.background(_:)`; if so, root page background can remain light-token only until a later framework-level or theme-level solution.
- Without a visible toggle, dark mode validation depends on generated CSS and Raptor's automatic color scheme script rather than manual UI interaction.
- Exact color tuning remains visual and may need one browser review pass after implementation.
- CSS duplication from Raptor style generation remains out of scope.
